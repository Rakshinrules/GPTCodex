====================================
1. СУТЬ ПРОЕКТА
====================================

Проект называется ПРОМТМЕЙСТЕР.

Это не просто генератор текста, не просто конструктор промтов и не просто чат с ИИ.
Это система настройки ИИ как профессионального помощника через:
- область,
- специализацию,
- роль,
- поведение,
- контекст,
- ограничения,
- формат результата,
- конфигурацию,
- библиотеку режимов,
- базу заготовок и шаблонов.

Ключевая идея проекта:
ПРОМТМЕЙСТЕР создаёт режим работы ИИ, а не просто один ответ.

Внутреннее краткое определение:
“Создаёт вайб-кодинг там, где его нет.”

Во внешнем нормальном определении:
Это инструмент сборки рабочей конфигурации ИИ под конкретную задачу.

Из concept.html следует, что центральный объект продукта — конфигурация ИИ, а не разовый текст. Из plan.html следует, что проект строится как продукт с ядром, UX, frontend, prompt engine, библиотеками и дальнейшим backend/AI слоем. Это сохраняется как базовая истина проекта.

====================================
2. ГЛАВНЫЕ ПРИНЦИПЫ, КОТОРЫЕ НЕЛЬЗЯ ТЕРЯТЬ
====================================

1. Source of truth важнее памяти чата.
Если в новом чате модель “помнит что-то иначе”, верить надо файлам, а не её импровизации.

2. Deterministic-first.
На текущем этапе база проекта детерминированная.
Сначала правила, flow, конфиги, prompt assembly, local storage, каскады.
Потом точечное усиление через API / AI.
Не наоборот.

3. Mobile-first.
Пользователь работает со смартфона.
Нельзя плодить перегруженные формы и длинные полотна.
Чипсы, короткие выборы, каскад, пошаговый flow — предпочтительны.

4. Не ломать рабочее ядро.
Уже были случаи, когда при добавлении нового слоя ломался JS, исчезали чипсы, разваливался первый экран.
Повторять это нельзя.

5. Сначала каркас, потом логика.
Если вводится новый слой (например modes), сначала сделать каркас, который не ломает базу.
Потом отдельно наращивать бизнес-логику.

6. Не самовольничать.
Нельзя:
- вносить несогласованные архитектурные развороты,
- подменять продукт “болтливым советчиком”,
- смешивать слои,
- перепрыгивать на backend, если не зафиксирован frontend layer,
- выдавать “улучшения” ради красоты, если рушится ясность.

7. Честность обязательна.
Нельзя говорить, что что-то сделано, если оно не проверено.
Нельзя говорить, что UI изменился, если это не проверено.
Нельзя врать про фикс багов.
Если есть сомнение — писать, что сомнение есть.

3. ПРОФИЛЬ ПОЛЬЗОВАТЕЛЯ И ОЖИДАНИЯ ОТ АССИСТЕНТА
====================================

Пользователь:
- Алексей
- практик
- мыслит системно
- в вайб-кодинге новичок, но быстро учится

Как ассистент должен себя вести:

- без расползания мысли,
- без “как обычно чат-бот”.

Как ассистент должен относиться к коду:
- код = рабочий инструмент, не украшение,
- сначала устойчивость, потом полировка,
- если версия загрязнилась — лучше clean version, чем бесконечные патчи,
- не ломать id, привязки HTML↔JS, flow, local storage,
- по возможности проверять синтаксис JS после правок.

8. API BLOCK — ЭТО ОТДЕЛЬНЫЙ ПРОДУКТОВЫЙ СЛОЙ
====================================

API нужен.
Но не как свободный “советчик”, не как чат, не как болтовня.

Его основное назначение:
1. Подбирать варианты.
2. Превращать их в чипсы.
3. Если вариантов нет — подсказывать, что именно писать в поле и как сузить запрос.

То есть правильная роль API:
- generator of options
- chips generator
- field filling assistant

А не:
- “я сейчас дам тебе рассуждение на тему”.

Правильная логика API в приложении:
- пользователь находится в конкретном поле / шаге,
- deterministic system знает контекст,
- API получает узкий запрос,
- API возвращает короткие релевантные варианты,
- UI превращает их в чипсы,
- если уверенных вариантов нет — API помогает текстом “как лучше заполнить поле”.

Нельзя:
- делать длинные ответы от API в UI,
- подменять flow чатом,
- размазывать продукт.

На данный момент:
- API существует как идея и внешний ресурс,
- прямой browser-call признан плохим и нестабильным,
- ключ в HTML хранить нельзя,
- в будущем ключ должен лежать на сервере, лучше в .env вне публичной папки,
- но это следующий этап, не сейчас.

Следующий шаг по API:
спроектировать API interaction logic block отдельно, после modes.
